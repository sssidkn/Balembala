# Notification System: требования и домены

## Бизнес-контекст

**Notification System** — универсальная система массовой и триггерной рассылки уведомлений по e‑mail, предназначенная для оперативного оповещения пользователей и интеграции с внешними системами-инициаторами. Доставка выполняется асинхронно через брокер сообщений, с повторными попытками, прозрачными статусами и механизмом отбраковки безнадёжных сообщений.

**Области применения (примеры):**
- ИТ и корпоративные процессы: инциденты/аварии, плановые работы, сервисные сообщения.
- Транзакционные уведомления: статус заказа/доставки/оплаты, OTP/подтверждения.
- Подписки и уведомления по событиям: изменения в профиле, подписка на контент, новости продукта.
- Образование/внутренние коммуникации: расписание, дедлайны, объявления.
- Публичные оповещения: предупреждения о событиях/ЧС.

**Основные пользователи:**
- Администратор/оператор (управляет контактами, шаблонами, запускает рассылки).
- Система‑инициатор (внешняя система, которая триггерит рассылку через API).
- Получатели уведомлений.

**Критерии успеха:**
- Быстрый запуск рассылок по шаблонам и группам контактов.
- Надёжная доставка: retry/DLQ, отсутствие потерь и контроль дублей.
- Прозрачность процесса: статусы обработки (queued/retry/dlq), логи.
- Безопасность и изоляция данных пользователей.

**Ограничения/допущения:**
- Канал доставки в MVP — **e‑mail через SMTP** (с перспективой расширения на SMS/Push/мессенджеры).
- Используется **обычный SMTP‑аккаунт общего назначения**, а не специализированный сервис рассылок (ESP). Поэтому действуют лимиты провайдера (антиспам, rate limiting), возможны временные ошибки и необходимость повторных попыток.
- Контакты и шаблоны принадлежат конкретному пользователю.

## Бизнес-требования

| ID | Бизнес-требование | Ценность |
|---|---|---|
| BR-1 | Система должна обеспечивать массовую и триггерную рассылку уведомлений по группам контактов | Быстрое информирование целевых групп |
| BR-2 | Оператор должен управлять контактами и шаблонами сообщений | Переиспользование контента и централизованное управление |
| BR-3 | Рассылка должна быть устойчивой к сбоям и обеспечивать повторные попытки доставки | Повышение доставляемости и снижение потерь |
| BR-4 | Должен быть механизм отбраковки безнадёжных сообщений (DLQ) | Разбор проблемных сообщений без зацикливания |
| BR-5 | Должна быть авторизация/изоляция данных между пользователями | Безопасность и мульти‑тенантность |
| BR-6 | Должна быть наблюдаемость процесса рассылки (статусы/логи) | Контроль выполнения и диагностика |
| BR-7 | Система должна интегрироваться по API с внешними системами-инициаторами | Автоматизация и снижение ручного труда |
| BR-8 | Система должна быть расширяемой по каналам доставки | Возможность добавлять SMS/Push/мессенджеры без переписывания ядра |

## Функциональные требования

### Аутентификация

| ID | User Story | Acceptance Criteria |
|---|---|---|
| FR-AUTH-1 | Как пользователь, я хочу зарегистрироваться, чтобы получить доступ к системе | `POST /api/register` возвращает JWT |
| FR-AUTH-2 | Как пользователь, я хочу войти, чтобы работать с контактами/шаблонами/рассылками | `POST /api/login` возвращает JWT |
| FR-AUTH-3 | Как gateway, я хочу валидировать токен, чтобы проксировать запросы дальше | `POST /api/validate-token` возвращает признак валидности и `user_id` |

### Контакты и шаблоны

| ID | User Story | Acceptance Criteria |
|---|---|---|
| FR-CT-1 | Как пользователь, я хочу создать шаблон, чтобы переиспользовать текст уведомления | `POST /api/templates` создаёт `template(id,title,message)` |
| FR-CT-2 | Как пользователь, я хочу редактировать/удалять шаблон | `PUT/DELETE /api/template/{id}` |
| FR-CT-3 | Как пользователь, я хочу просмотреть список своих шаблонов | `GET /api/templates` |
| FR-CT-4 | Как пользователь, я хочу создать/редактировать/удалять контакт | `POST/PUT/DELETE` для `contact` |
| FR-CT-5 | Как пользователь, я хочу привязать набор контактов к шаблону (группа получателей), чтобы удобно запускать рассылку | `POST /api/template/{template_id}` с `contacts_id[]` |
| FR-CT-6 | Как пользователь, я хочу получить контакты, связанные с шаблоном | `GET /api/contacts/{template_id}` |

### Инициация рассылки

| ID | User Story | Acceptance Criteria |
|---|---|---|
| FR-N-1 | Как пользователь/внешняя система, я хочу инициировать отправку уведомления по шаблону, чтобы система начала рассылку адресатам | `POST /api/send/{templateId}` возвращает статус принятия |
| FR-N-2 | Система должна публиковать задание на отправку в брокер сообщений | Сообщение появляется в `notifications` topic |
| FR-N-3 | Система должна формировать корреляцию рассылки (correlationId/operationId), чтобы отслеживать обработку | Идентификатор сохраняется в сообщении и логах |

### Доставка

| ID | User Story | Acceptance Criteria |
|---|---|---|
| FR-S-1 | Как sender‑сервис, я хочу читать задания из `notifications`, чтобы отправлять письма | Консьюмер читает и шлёт через SMTP |
| FR-S-2 | Как система, я хочу при неуспехе отправки помещать сообщение в `retry`, чтобы повторить отправку | Ошибка → `retry` topic |
| FR-R-1 | Как rebalancer‑сервис, я хочу выполнять повторные попытки с лимитом, чтобы повышать доставляемость | До `maxAttempts` → обратно в `notifications` |
| FR-R-2 | Как система, я хочу помещать безнадёжные сообщения в `dlq`, чтобы их можно было разбирать | Превышен лимит → `dlq` |

## Нефункциональные требования

| ID | НФТ | Целевое значение |
|---|---|---|
| NFR-AV-1 | Доступность API (gateway + основные сервисы) | ≥ 99.5% в месяц |
| NFR-PERF-1 | Время ответа API на CRUD (контакты/шаблоны) | p95 ≤ 300 мс |
| NFR-PERF-2 | Пропускная способность отправки (MVP, обычный SMTP‑аккаунт) | **до 100 писем/мин** при стабильном SMTP; при лимитах провайдера допускается рост времени доставки и рост доли retry |
| NFR-REL-1 | Устойчивость доставки | Наличие `retry` + `DLQ`; отсутствие потерь при падении `sender` (за счёт Kafka) |
| NFR-SEC-1 | Безопасность | JWT; изоляция данных по `user_id` |
| NFR-OBS-1 | Наблюдаемость | структурные логи (минимум), метрики/трейсы — опционально |
| NFR-SCALE-1 | Масштабирование | Sender/Rebalancer горизонтально; Kafka масштабируется партициями/консьюмерами |
| NFR-EXT-1 | Расширяемость по каналам | Архитектура допускает добавление новых “adapters” доставки без изменения ядра инициации |

## Домены

### Бизнес-домены

| Бизнес-домен | Что делает |
|---|---|
| Управление пользователями и доступом | регистрация, логин, авторизация запросов |
| Управление адресатами и контентом | контакты, шаблоны сообщений, группы/привязки |
| Управление рассылками | запуск рассылки, маршрутизация, постановка задач |
| Доставка и надёжность | фактическая отправка, retry, DLQ |
| Интеграция и API‑периметр | единая точка входа, проксирование, политика безопасности |
| Расширение каналов | подключение дополнительных каналов (SMS/Push/мессенджеры) |

### Прикладные домены (и реализация)

| Бизнес-домен | Прикладные домены | Реализация |
|---|---|---|
| Управление пользователями и доступом | JWT, регистрация/логин, валидация токена | `auth-service` |
| Интеграция и API‑периметр | роутинг, проверка токена перед проксированием, единый порт входа | `api-gateway (nginx)` |
| Управление адресатами и контентом | CRUD шаблонов, CRUD контактов, связь шаблон↔контакты | `contacts-templates-service` |
| Управление рассылками | принять команду “send”, сформировать сообщение в брокер | `notifications-service` |
| Доставка и надёжность | консьюм из Kafka, SMTP отправка, отправка в retry | `sender-service` |
| Доставка и надёжность | обработка retry, лимит попыток, перевод в DLQ | `rebalancer-service` |
| Сообщения/очереди | топики, партиции, группы консьюмеров, репликация | `Kafka` |
| Расширение каналов | интерфейс “Channel Adapter” + отдельные сервисы-адаптеры | (планируемое расширение) |

## Требования к веб-интерфейсу (Frontend)

### Роли и доступ

- Пользователь взаимодействует с системой через веб-приложение после входа.
- Все операции в UI выполняются в контексте текущего пользователя (`user_id` определяется сервером по JWT).

### Экранные сценарии (Screen-flow)

1. **Login / Register**
   - Ввод email/пароля, регистрация нового пользователя.
   - После успешного входа — переход в Dashboard.

2. **Dashboard**
   - Быстрые действия: создать шаблон, добавить контакт, запустить рассылку.
   - Блок “Последние операции” (минимум: список последних запусков/лог-отображение).

3. **Templates (Шаблоны)**
   - Список шаблонов.
   - Создание/редактирование/удаление шаблона.
   - Просмотр деталей шаблона (текст сообщения).

4. **Contacts (Контакты)**
   - Список контактов.
   - Создание/редактирование/удаление контакта.
   - Поиск/фильтрация по имени/email (минимум: поиск по строке).

5. **Template ↔ Contacts (Привязки)**
   - На странице шаблона: выбор контактов (multi-select) и сохранение привязки.
   - Просмотр привязанных контактов и возможность удалить связь.

6. **Send Notification (Запуск рассылки)**
   - Выбор шаблона.
   - Подтверждение отправки (диалог подтверждения).
   - Отображение результата: “принято в обработку” + идентификатор/время.

7. **Status / Logs (Статусы/логи) — опционально в MVP**
   - Просмотр агрегированного статуса отправок (успех/ошибка/retry/dlq) или хотя бы лог-лента.

### Функциональные требования (Frontend)

| ID | User Story | Acceptance Criteria |
|---|---|---|
| FR-FE-1 | Как пользователь, я хочу зарегистрироваться/войти через веб-интерфейс, чтобы получить доступ к функциям системы | Формы login/register; после успеха сохраняется токен; доступ к защищённым страницам |
| FR-FE-2 | Как пользователь, я хочу видеть список шаблонов и управлять ими (CRUD), чтобы готовить уведомления | UI-страница Templates; операции вызывают API; список обновляется |
| FR-FE-3 | Как пользователь, я хочу видеть список контактов и управлять ими (CRUD), чтобы управлять адресатами | UI-страница Contacts; операции вызывают API; список обновляется |
| FR-FE-4 | Как пользователь, я хочу связывать контакты с шаблоном, чтобы рассылка шла по группе | Multi-select контактов на странице шаблона; сохранение привязки через API |
| FR-FE-5 | Как пользователь, я хочу запускать рассылку из веб-интерфейса, чтобы быстро отправлять уведомления | Кнопка “Send”; запрос `POST /api/send/{templateId}`; показ “queued/accepted” |
| FR-FE-6 | Как пользователь, я хочу видеть подтверждение/ошибку после действия, чтобы понимать результат | Toast/alert; обработка 4xx/5xx с понятным сообщением |
| FR-FE-7 | Как пользователь, я хочу искать контакты/шаблоны, чтобы быстро находить нужное | Поиск по строке |
| FR-FE-8 | Как пользователь, я хочу выйти из системы, чтобы завершить сессию | Logout очищает токен и возвращает на Login |

### Нефункциональные требования (Frontend)

| ID | НФТ | Целевое значение |
|---|---|---|
| NFR-FE-1 | Юзабилити | Основные сценарии (CRUD, send) выполняются ≤ 3 клика от Dashboard |
| NFR-FE-2 | Производительность UI | First Contentful Paint ≤ 2.5 сек, p95 реакции UI на действия ≤ 200 мс |
| NFR-FE-3 | Надёжность | Корректная обработка ошибок API (показ сообщения, отсутствие “пустых” экранов) |
| NFR-FE-4 | Безопасность | Токен не логируется; защищённые маршруты недоступны без авторизации; ввод валидируется на клиенте |
